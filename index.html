Agenda
1 背景 & 動機
核心問題：排序粒度太粗
  • 目前以 adset（廣告組） 為最小排序單位，一個 adset 下有多個 creative
  • ML model 預測 adset-level CTR — 無法區分同 adset 內不同 creative 的品質
  • Creative 選擇靠 UCB bandit / 隨機（Phase 14），沒有 ML model 參與
  • Adset embedding = 所有 creative embedding 的平均，個別 creative 語意被稀釋
目標 & 預期效益
  • CPM 穩定化 — creative-level CTR 預測更精準，出價與實際效果一致
  • 推薦更精準 — 針對不同 user 推送最合適的 creative，提升 CTR
  • 廣告主價值 — 好素材得到更多曝光，差素材自然被淘汰
早期展開（方案 A）的核心概念
  • Matching 後，在 Phase 2.5 將每個 adset 展開為 N 個 (adset_id, creative_id) candidates
  • 後續 所有 phase（Feature Loading → CTR → Scoring → Ranking）都以 creative 為單位操作
  • 每個 creative 有自己的 CTR 預測，直接參與排序 & 定價
  • 排序品質最佳，但改動範圍最大

3 改動全景圖
Ranking Pipeline — 按 Phase 順序
Phase 1：Matching 不需改動
  • 功能：根據 user context 從 RediSearch 檢索符合 targeting 條件的 adset candidates
  • Contextual Segments：從 MongoDB 取得 post_tags + forum_tags，合併為 contextual_segments 用於 targeting 比對
  • RediSearch Query：組合 layout、ad_type、gender、age range、region、school、platform 等 targeting 條件；進階 targeting（forums / tags / segments / custom audience / keywords / interest_l2_tags）以 OR 方式匹配
  • Realtime Targeting：額外查詢 is_realtime_targeting=True 的 candidates（非 sex request）
  • Feature Loading：從 RediSearch Feature Index + Bigtable 載入每個 adset 的 feature（bidding、budget、creative IDs、tags 等），包裝為 AdsetWrapper
  • Cache：候選結果 Redis cache TTL 600s，按 (member_id, platform, layout, types, post_id/forum, region) 分 key
不需改動原因：Matching 維持 adset-level 檢索即可，creative 展開在 Phase 2.5 進行。RediSearch index 不需要改為 creative-level。
matching.py redisearch_query_builder.py adset_candidate.py adset_loader.py
Phase 2：Ad Filter 不需改動
  • 功能：過濾不符合投放條件的 adset 和 creative，產出 filtered_available_creative_ids
  • Layout Check（pre-filter）：僅保留支援當前 contextual_layout 的 adsets
  • Creative Filter：對每個 adset 內的 creatives 做去重過濾，移除最近曝光過的 creative_dedup_id；全部被過濾的 adset 整個移除
  • 輸出：每個 adset 帶有 filtered_available_creative_ids（存活的 creative 列表）和 filtered_creative_dedup_ids
Ad Filter 篩選條件明細
Hard Filter — Adset 層級（按 enum 順序執行）
  1. Takeover：版位類型互斥（takeover adset ↔ takeover unit）
  2. Experiment Budget：exp_cost ≥ daily_budget × exp% × 1.03（Local Redis cache TTL 1h）
  3. Daily Budget：today_cost ≥ daily_budget × 1000（Local Redis cache TTL 1h）
  4. Impression Cap：today_imp ≥ daily_cap（預設 3，Remote Redis TTL 30min）
  5. ADN Budget Pacing：ADN unit 按分鐘 pro-rata × 0.7 控速
Hard Filter — Creative 層級（僅 TW iOS/Android）
  • Recent Creative Exposure：同 layout 素材去重（Redis hset TTL 1hr，sex=5min）；全部過濾則移除 adset
Soft Filter（全部被過濾則不執行）
  • Recent Adset Exposure：避免短時間重複曝光（Redis hset TTL 60s）
平台差異
  • mobileWeb / Desktop：僅 Daily Budget + Takeover，無 creative 去重
  • Takeover unit：僅 Daily Budget + Impression Cap + Takeover
  • Sex request：移除 Exp Budget；creative exposure TTL 降為 5min
不需改動原因：Ad Filter 已有 creative-level 過濾（RecentLayoutDedupCreativeExposureExcludeRule）。Creative 展開（Phase 2.5）使用 filter 產出的 filtered_available_creative_ids，流程自然銜接。

Phase 2.5：Creative 展開 ⚡ 新增 中
  • 新增步驟：在 Filter 之後、Load Ranking Data 之前，插入 creative 展開邏輯
  • 展開邏輯：每個 adset 根據 filtered_available_creative_ids（Phase 2 產出）展開為 N 個 (adset_id, creative_id) candidate
  • Candidate 數量：平均 ×2.8（每個 adset 平均 2.8 個存活 creative），後續所有 phase 操作對象從 adset → creative candidate
  • 資料結構改變：目前 available_adsets: List[AdsetWrapper] 每個 wrapper 內含 creatives[] nested list；改為攤平的 candidate list，每個 candidate 帶自己的 creative_id
  • Dedup ID：creative_dedup_id 在展開時從 RediSearch 帶入即可（資料在 adset 載入階段已存在於 creative_dedup_ids 欄位，Phase 14 只是將被選中 creative 的 dedup_id 寫回 adset-level 欄位）

Phase 3：Load Ranking Data 高
  • 功能：從 Bigtable 載入每個 adset 的排序用 features（CTR、VTR、impression 統計、tag 表現等）
  • Row Key：ad_set_id#<id>#layout#<layout>，column family = adsets
  • Adset Features：lifetime CTR/impression、past 72h CTR（整體 + male/female）、3d CTR/clicks/views、smoothed CTR、VTR dicts（platform×gender×age 多層級）、budget goal 相關、tag performance（9 欄位）、context post tag overlap（12 欄位）、viewed post similarity
  • Creative Features：nested 在 adset row 內的 creatives[] array，包含 creative_id、lifetime impression/click、past_72_ctr（整體 + gender）、smoothed/dynamic CTR 等
  • Sampling Metadata：IS_ADSET_COLD_START（lifetime_imp < 1000）、IS_DAILY_COST_UNREACHED（today_cost < 30K）、IS_SAMPLED
  • Cache：Redis pipeline cache TTL 900s（YOUJI_RANKING_FEATURE_CACHE_V3）
  • Fallback：Bigtable 異常時返回最小結構（僅含 adset_id + creative_ids list），其餘 features 省略
Creative Ranking 需改動
  • Bigtable fetch 需 deduplicate（同 adset 的多個 candidate 只查一次 row）再 broadcast features 到 N 個 candidate
  • 目前 creative features 打包在 adset row 的 creatives[] array 裡，需改為獨立 creative-keyed row：creative_id#<id>#layout#<layout>

Phase 4：Recall 低

  • 正常路徑：impression_guarantee（KA/BD 保量）adsets + GT warm（非 cold start 且非 sampled）adsets 合併為 recalled_adsets
  • Sampling 路徑：若 should_apply_sampling_logic=True 且有 is_sampled=True 的 adsets，shuffle 後回傳並標記 is_sampled_request=True（由 feeder 據此跳過 build feature、CTR、ranking 等後續 phase）；若 sampled 為空則 fallback 到正常路徑
  • Cold Start 歸屬：cold start 且 is_sampled=True 的 adset 走 sampling 路徑；cold start 但 is_sampled=False 的 adset 因不符合 GT warm 條件而被排除
  • maximum_returns=2：傳入但未使用（歷史遺留），實際截斷在 Phase 13 Top-K

Phase 5：Build Features 高

功能：從不同來源取出 5 大類 features，組裝成統一的 model input dict（per candidate）

  1. Member Features（共用，純取出）
    • member_info.to_model_dict() — demographics（age, gender, region）+ behavior（feed/dice 7d 指標）+ member embeddings
  2. Contextual Features（共用，純取出）
    • 從 ad_request 取 platform, hour, unit_id, layout
  3. Embedding Features（per-adset，純取出）
    • AdsetEmbeddingFeatureRepository.loads() 從 Bigtable 取 512-dim adset embedding（Redis cache TTL 900s）
  4. Adset Features（per-adset，取出 + 計算）
    • adset.to_dict(model_used=True) 取出 ~53 個 compiled features：
    • Basic：CTR / VTR / impression 各維度
    • Creative：展開每個 creative 的 CTR（含 gender split）、CLICK_VIEWS_INFO_SCORE → 計算 creative_3d_max_ctr_v2 = max(creative CTR) + smoothed 版本
    • Adset Lifetime：adset_lifetime_ctr（fallback ctr_serving）
    • Identity / Budget / Pacing / Targeting
    • Tag Performance（9）/ Context Tag Overlap（12）/ Similarity
    • Smoothed CTR Fallback：仍為 -1 時，取 account_type 級別 fallback（floor 0.0001）
    • Cascading VTR：platform×gender×age → gender → platform → overall，取第一個非 -1 值
  5. Interaction Features（per member × adset，取出 + 計算）
    • InteractionModelFeatureRepository.loads() — 用 member features 和 adset 的 forum/adset_id 交叉查找：
    • Forum：forum_ctr / forum_group_ctr / forum_visit_count / forum_is_subscribed
    • Engagement：member_adset 1d/3d/7d views/dwells/clicks（7 欄位）
    • Tag：match count/rate, L1/L2 CTR/views, interest scores（13 欄位）

組裝：主迴圈 for each adset，合併 {member + contextual + interaction[adset_id] + adset_features + embedding} → List[Dict]

Creative Ranking 需改動

A. Embedding Features（對應 3）

目前 repository.loads(adset_ids) 回傳 List[List[float]]，以 index 對齊 adsets 順序

  • len(embeddings) != len(candidates) 斷言失敗 → 改用 dict lookup（同 adset 多 candidate 會重複查同一份 embedding）
  • 擴展 CreativeEmbeddingFeatureRepository（目前已用於 member embedding 載入 last_clicked_creative_embedding），改為同時以 creative_id 作為 key 載入 ad candidate 的 creative-level embedding
  • avg_adset_embedding → creative_embedding（512-dim，從 adset 平均 embedding 改為個別 creative embedding）。注意：avg_adset_embedding 目前仍是 production 唯一的 ad-side embedding input（training + serving），切換需同步更新 predictor.yaml 和訓練 pipeline

B. Adset Features（對應 4）

目前 adset.to_dict(model_used=True) 回傳 adset-level compiled features（~53 個）

  • 8 個 creative_3d_max_ctr 從 max() aggregation → per-creative 值（含 smoothed × gender variants），需改 _creative_max_ctr_fn() 的計算邏輯
  • adset.to_dict() 目前只回傳 adset-level features → 需額外輸出 per-creative compiled features（如該 creative 自己的 CTR），讓每個 candidate 拿到對應 creative 的值

C. Interaction Features（對應 5）

目前 InteractionModelFeatureRepository.loads() 回傳 Dict[adset_id, features]，每個 adset 計算一次

  • interaction_features[adset_id] 同 adset 展開的多個 creative candidate 需 broadcast 同一份 interaction features（member × adset 層級不因 creative 不同而變），避免重複計算

D. 組裝

目前 for i, adset in enumerate(adsets) 一個 adset 產出一個 feature dict

  • 迴圈結構從 per-adset → per-(adset, creative) pair：一個 adset 產出多個 feature dicts，index 對齊方式全部要跟著改（avg_adset_embeddings[i]、interaction_features[adset_id]、adset_model_features）

Phase 7：Estimated VTR 不需改動
  • 功能：為 CPV（Cost-Per-View）bidding 的 adset 設定 VTR，用於計算 eCPM
  • Rule-based：不使用 ML model，直接取 Phase 5 計算好的 video_views_coalesced_age_gender_platform_adset_vtr（cascading VTR）
  • 適用條件：僅 bidding_strategy == CPV 的 adsets；非 CPV adset 的 estimated_vtr 保持 None
  • VTR Cascade：platform×gender×age → platform×gender → platform-only → overall past_72_adset_vtr_v5，取第一個非 -1 值
不需改動原因：VTR 目前為 adset-level rule-based（無 per-creative VTR），且 CPV 佔比低。Creative ranking 初期可維持 adset-level VTR，後續視需求再做 per-creative VTR。

Phase 8：Score Threshold + Quality Filter 不需改動
  • Score Threshold：針對 BD_EVENT adsets + list layout，當 budget_burn_rate > 0.8 時，移除 model_prediction_score < threshold 的 adset。保留 1000/daily_impressions 機率隨機 bypass（確保最低曝光量供資料收集）
  • Quality Filter：依 skip rate 將 adset 分為 skipped（保留，用於 sampling）與 non-skipped（進入品質過濾）兩組：
    • Skip rate：min(max(500/total_impressions, 0.05), 0.50)，同一 request 共用一個 random roll
    • Percentile-based filtering：non-skipped adsets 根據 quality_percentile（Redis cache，基於 overspending 狀態動態更新）查找 p{percentile} threshold，estimated_ctr < threshold 的 adset 被移除
    • DAY_END_HOUR：日末時段所有 adset 標記 skipped，跳過品質過濾
不需改動原因：Score threshold 僅針對 BD_EVENT，config 為 adset-level，per-creative score 自然適用。Quality filter 的 _update_percentiles 雖以 adset_id 為 key，但有 Redis lock（60s TTL）保護，同一 request 內多個 creatives 只會更新一次，不需 dedup。Percentile threshold 過濾 per-creative estimated_ctr 語意合理。

Phase 9：Ranking Price 不需改動
  • 功能：計算每個 adset 的 eCPM（effective CPM），作為排序基礎價格
  • eCPM 公式：
    • CPC：eCPM = estimated_ctr × bidding_price × 1000
    • CPV：eCPM = estimated_vtr × bidding_price × 1000
    • CPM：eCPM = bidding_price（直接使用出價）
    • Bigtable fallback adsets：eCPM = reserve_price[ad_type]
  • Reserve Price Filter：ranking_price < reserve_price[ad_type] 的 adset 被移除（底價過濾，底價從 MongoDB 載入）
不需改動原因：eCPM 公式中 estimated_ctr 在上游已是 per-candidate，bidding_price 為 adset-level 共享，公式自然適用。Reserve price filter 與 Bigtable fallback 皆以 adset_id 判斷，candidate 繼承 adset_id，邏輯不變。
ranking_price.py
Phase 10：Targeting Score 不需改動
  • 功能：為使用 interest-based targeting 的 adset 加分，提升排序優先順序

Phase 11：Cost Price（2nd-price auction） 高
  • 功能：執行 2nd-price auction 決定每個 adset 的實際扣費價格
  • 2nd Price 計算：按 ranking_price 排序後反向遍歷，每個 adset 的 second_price = 下一個不同 account 的 ranking_price（避免 self-bidding）
  • 1st Price 例外：is_impression_guarantee 或 GT + CPM 使用 1st price（直接用自己的 ranking_price）
  • Price Floor：以 price_floor_probability 機率取 max(cost_price, floor_cost_price)；特殊帳號 80%、sex 30%、一般 50%
  • Raise Price：hardcode config 可對特定 adset + layout 加價（不超過 ranking_price）
Creative Ranking 需改動
  • 核心問題：同 adset 的 creative 互相計算 2nd price — 經濟學上不合理（自己跟自己競價）
  • 需 skip same-adset 比較：2nd price 應取「下一個不同 adset 且不同 account」的 ranking_price
cost_price.py
Phase 12：Adset Ranking → Creative Ranking 中
  • 功能：計算 additive ranking score 並排序，決定最終 creative candidate 優先順序（改動後排序對象從 adset → creative）
  • Ranking Score 公式：2.0 × ctr_score + 1.0 × pacing_bid_score + 1.0 × tag_targeting_score
    • ctr_score：CPC/CPM → min-max normalize(log(CTR))；CPV → min-max normalize(VTR)
    • pacing_bid_score：base_bid × pacing_factor，其中 pacing_factor = 1 + scale × tanh(sensitivity × diff_ratio)，diff_ratio 反映 budget 花費進度（under/over spending）
    • tag_targeting_score：1.0 或 20.0（見 Phase 10）
  • Budget Ratio：target_ratio = elapsed_hourly_budget / current_budget、current_ratio = today_cost / current_budget；GT 帳號和 zero_cost 不計算
  • Hour 23 特殊邏輯：guaranteed adsets 按 ka_bd_priority 排序（KA 保量優先）+ 其餘按 ranking_price 排序，合併後回傳
  • CTR Rank Metadata：記錄每個 adset 的 estimated_ctr_rank 和 estimated_ctr_diff（與最高 CTR 的差距），用於 logging
Creative Ranking 需改動
  • 排序對象從 adset → creative candidate
  • Budget ratio 需 memoize per adset（避免同 adset N 個 candidate 重複計算 today_cost / current_budget）
  • 同 adset 的 creative 可能分散在不同排名位置 → 排序後加 per-adset dedup（每 adset 只保留最高分 creative），確保 Top-1 回傳不同 adset 的結果
adset_ranking.py ranking_metric.py
Phase 13：Top-K 移除
  • 目前功能：從排序後的 adset list 取前 K 個（maximum_returns = 2），即 adsets[:2]，供 Phase 14 Creative Ranking 使用
  • 移除原因：Top-K 存在的目的是縮小候選範圍給 Phase 14 做 per-adset creative 選擇。Phase 14 移除後，不再需要中間的 Top-K 步驟，Phase 12 排序完直接由 Phase 15 取 Top-1 即可
top_k.py
Phase 14：Creative Ranking（UCB bandit） 移除
  • 目前功能：在排好序的 Top-K adset 中，為每個 adset 選出最佳 creative
  • Scoring 分流（creative_scorer.py:74-87）：依 account_type + is_sampled 決定 creative 選擇策略
    • UCB bandit：account_type 為 bd_event 或 gt，且 is_sampled=0
    • Random：account_type 為 ka，或任何 is_sampled=1 的 adset
  • Account Type 對應（constants/__init__.py）：campaign_goal → account_type
    • guaranteeDeal / takeover → KA（隨機）
    • project → BD_EVENT（UCB）
    • awareness / traffic → GT（UCB）
  • Random 機制（KA / sampled）：每個 creative 賦予 random.random()（均勻分佈 0~1）作為 bandit_score，完全不參考歷史 CTR 表現
  • UCB 公式（BD_EVENT / GT）：score = decay × dynamic_smoothed_72h_ctr + (1-decay) × √(2·ln(adset_imp)/creative_imp)
    • decay = min(creative_imp, 2850) / 3000：creative 曝光越多，exploitation 權重越高
    • Exploration term（UCB1）：adset 曝光多、creative 曝光少 → 探索分數高
  • 選擇：每個 adset 的 creative_ranking_states 按 bandit_score 降序排列，取 [0] 設定 adset.creative_id 和 creative_dedup_id

Creative vs. Adset Feature Granularity Trade-off
核心問題不是 Creative Signal 不足，而是 Granularity vs. Stability 的權衡：
Creative-level Adset-level
Granularity ✓ 能區分同 adset 下不同素材 ✗ 同 adset 素材共用同一組值
Stability ✗ 資料量少、variance 高、cold start 嚴重 ✓ 聚合後統計穩定
建議策略：Hierarchical Features
  • 兩層共存：同類 feature（如 CTR）同時提供 adset-level 與 creative-level，讓 model 自行學習權重 — 資料充足時依賴 creative-level，稀疏時自動 fallback 到 adset-level
  • Smoothing + 信賴度：creative 統計值 shrink toward adset-level（Empirical Bayes），並加入 impression_count 讓 model 判斷信賴度

風險影響
● 推論延遲增加 P99 latency 超標（×2.8 avg，P99 ×14）
● Creative cold start 現行 cold start 判斷是 adset-level（adset_lifetime_impression < 1000），新 creative 掛在 warm adset 下不會被識別為 cold start，長期無法獲曝光

● Phase 14 Random 機制遷移 KA / sampled adset 的 creative 失去隨機探索，exploitation-only 導致 creative 枯竭

● A/B 雙模式共存 兩組共用 budget 計數器（pacing 互相干擾）、歷史 CTR feature（回饋迴圈）、dedup cache（creative 選擇交叉）；同時部署雙 model 增加 MLServer 負載；pipeline 至少 7 處分支
